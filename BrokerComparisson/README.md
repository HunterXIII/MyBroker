# СРАВНИТЕЛЬНЫЙ АНАЛИЗ ПРОИЗВОДИТЕЛЬНОСТИ RABBITMQ И REDIS

## 1. ОПИСАНИЕ ЭКСПЕРИМЕНТА
**Цель:** Сравнить пропускную способность и задержки (latency) брокеров сообщений в условиях ограниченных системных ресурсов.

**Стенд тестирования:**
* **Язык:** Go (Producer/Consumer с использованием горутин).
* **Окружение:** Docker-контейнеры (Arch Linux).
* **Аппаратные лимиты (Docker Compose):** Для обоих контейнеров установлены жесткие ограничения, чтобы исключить влияние избыточных ресурсов хост-системы. Это позволяет увидеть, как ведут себя системы в условиях реального дефицита ресурсов.

**Настройки лимитов (deploy.resources.limits):**
* cpus: '1.0' (Максимум 1 ядро CPU)
* memory: 1G (Максимум 1 ГБ RAM)

## 3. ПОДРОБНЫЙ АНАЛИЗ

### Сценарий 1: Работа под экстремальной нагрузкой (Stress Test)
При попытке отправить 100 000 сообщений без ограничений по скорости, оба брокера упираются в лимит 1 CPU.
* **Redis** показывает лучшую пропускную способность (на ~30% выше), но задержки начинают расти до сотен миллисекунд из-за накопления очереди в памяти.
* **RabbitMQ** на мелких сообщениях (128 B) показал P95 в 2.5 секунды. Это указывает на высокую нагрузку на планировщик Erlang при попытке обеспечить гарантии доставки в условиях нехватки процессорного времени.

### Сценарий 2: Тяжелые сообщения (100 KB Payload)
Это наиболее показательный тест, демонстрирующий архитектурную разницу.
* Хотя **Redis** формально быстрее по RPS, его задержка P95 взлетела до 1.52 секунды. Работа с крупными объектами в рамках 1 ГБ выделенной памяти заставляет Redis тратить ресурсы на менеджмент памяти, что "бьет" по времени отклика.
* **RabbitMQ** справился со сценарием значительно стабильнее. Его P95 составил всего 476.7 мс. RabbitMQ лучше оптимизирован для потоковой передачи крупных пакетов данных, не допуская критического роста задержки.

### Сценарий 3: Фиксированный поток (Rate Limit 5000)
При ограничении входящего потока до 5000 msg/sec, ресурсов CPU хватает для своевременной обработки.
* Оба брокера переходят на микросекундный уровень задержек.
* **Redis** лидирует с показателем 699 µs против 820 µs у **RabbitMQ**. Это подтверждает превосходство Redis в сценариях, где важна минимальная задержка на легких данных.

### Сценарий 4: Высокая конкурентность (100 Workers)
Проверка работы при большом количестве одновременных TCP-соединений.
* RabbitMQ пострадал от переключения контекста (context switching) сильнее: P95 вырос до 2.38 с.
* Redis удержал более предсказуемую задержку в 1.33 с, подтверждая свою эффективность как легковесного брокера для множества параллельных клиентов.

## 4. ВЫВОДЫ ПО ДЕГРАДАЦИИ (SINGLE INSTANCE)

**1. Точка отказа Redis:**
Наступает при передаче крупных объектов (от 100 КБ) или заполнении RAM. Брокер начинает "захлебываться", и задержка для потребителя растет экспоненциально, так как Redis отдает приоритет хранению данных в памяти, а не скорости их вычитки при перегрузке.

**2. Точка отказа RabbitMQ:**
Наступает при взрывном росте количества воркеров на слабом CPU. Накладные расходы на протокол AMQP и поддержание очередей забирают ресурсы у логики доставки. Однако RabbitMQ лучше держит удар при работе с "тяжелым" контентом.

**3. Ресурсные лимиты:**
Ограничение в 1 CPU / 1 GB RAM является жестким порогом. В этих условиях Redis — спринтер (быстр на коротких дистанциях с мелкими данными), а RabbitMQ — стайер (медленнее, но надежнее держит темп при больших весах).

**Итоговая рекомендация:**
* Выбирайте **Redis Streams**, если вам нужны максимальные RPS и минимальные задержки на небольших сообщениях.
* Выбирайте **RabbitMQ**, если ваши сообщения весят более 100 КБ или если стабильность времени доставки (Latency) для вас важнее, чем пиковая пропускная способность.
