# ПРАКТИЧЕСКАЯ РАБОТА: СРАВНЕНИЕ RABBITMQ И REDIS КАК БРОКЕРОВ СООБЩЕНИЙ

## 1. ЦЕЛЬ И УСЛОВИЯ ЭКСПЕРИМЕНТА
**Цель:** Сравнить производительность, стабильность и потребление ресурсов RabbitMQ и Redis (Single Instance) в одинаковых условиях.

**Условия стенда:**
* Инструмент тестирования: Собственный producer и consumer на Go (с использованием горутин).
* Формат сообщений: Массив байт (128 B, 1 KB, 100 KB).
* Окружение: Docker-контейнеры на Arch Linux.

## 3. АНАЛИЗ РЕЗУЛЬТАТОВ ПО ЭТАПАМ

### Эксперимент 1: Базовое сравнение (Стресс-тест, 100 000 сообщений по 128 байт)
На максимальной скорости отправки без ограничений (Flooding):
* **Пропускная способность:** Redis показал более высокий Consumer RPS (7 402 против 4 945 у RabbitMQ). Благодаря In-Memory архитектуре, Redis быстрее принимает поток в память.
* **Задержка (Latency):** RabbitMQ продемонстрировал превосходную доставку в реальном времени — P95 составил всего 2.65 ms. В то же время у Redis задержка выросла до 1.16 s. Это связано с тем, что Redis быстро буферизирует данные, но при перегрузке консьюмер начинает вычитывать их большими пачками с задержкой.

### Эксперимент 2: Влияние размера сообщения (10 000 сообщений по 100 KB)
При увеличении Payload до 100 КБ оба брокера показали существенную деградацию производительности:
* Пропускная способность консьюмера RabbitMQ упала до 795 msg/sec, P95 Latency выросла до 117 ms.
* У Redis консьюмер обрабатывал 1 759 msg/sec, но задержка осталась на уровне около 1 секунды.
* **Критическое наблюдение (Деградация):** На тяжелых сообщениях Redis моментально "съел" 2.7 GiB оперативной памяти (92% CPU), так как хранит потоки в RAM. RabbitMQ в аналогичных условиях ведет себя стабильнее по памяти (~275 MB), активно сбрасывая данные на диск, что снижает скорость, но спасает систему от OOM (Out Of Memory).

### Эксперимент 3: Влияние интенсивности потока (Стабильная нагрузка, Rate 5000 msg/sec)
При дозированной нагрузке и размере 1 KB оба брокера показали идеальную работу:
* Latency у обоих брокеров упала до микросекунд (RabbitMQ: P95 = 784 µs, Redis: P95 = 663 µs).
* В контролируемых условиях Redis отрабатывает немного быстрее за счет отсутствия дисковых операций I/O, однако RabbitMQ показывает сопоставимые результаты.

## 4. ИТОГОВЫЕ ВЫВОДЫ

1. **Пропускная способность:** Redis Streams выигрывает по чистому количеству обрабатываемых сообщений в секунду, особенно на небольших Payload, благодаря работе в оперативной памяти.
2. **Влияние размера сообщения:** RabbitMQ лучше переносит большие сообщения (от 100 KB и выше). Несмотря на падение скорости, он не перегружает RAM. Redis на больших сообщениях стремительно деградирует по памяти (скачок до 2.7 GiB), что при длительной нагрузке приведет к падению (OOMKilled) Single Instance.
3. **Момент деградации Single Instance:** * **RabbitMQ** начинает деградировать при нехватке места на диске или при попытке отправить сотни тысяч сообщений без ограничений скорости (включается механизм Flow Control, блокирующий TCP-сокеты продюсеров).
   * **Redis** деградирует при исчерпании RAM. При агрессивной отправке он накапливает backlog, резко увеличивая Latency для конечного потребителя (до 1+ секунды).
4. **Инструментарий:** Разработанный стенд на Go (с воркерами и замерами P95 через пакет stats) оказался отличным выбором для гибкого тестирования. Он позволил точно отследить задержки и ограничения конкретных драйверов, избежав "черных ящиков" стандартных утилит вроде JMeter. 

**Краткий вердикт:** Для высокоскоростной аналитики и мелких ивентов, где не страшна потеря при перезагрузке, лучше подойдет Redis. Для гарантированной доставки крупных пакетов данных и стабильного потребления RAM (например, в финтехе или сложных энтерпрайз-системах) — RabbitMQ.
